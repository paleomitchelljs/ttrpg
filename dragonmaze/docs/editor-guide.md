# Zone editor вҖ” guide (Claude-facing)

A working map of the Dragon Maze tile editor, written so a future session can dive
into editor work without re-reading every file. For the *art* side (how a PNG
becomes a tile or a sprite) read [`art-pipeline.md`](art-pipeline.md) first вҖ” this
doc is about the editor that **places and paints** what that pipeline produces.

The editor is two tools in one page (`editor.html` + `editor.js`), talking to a
tiny dev server (`serve.mjs`) that shells out to the Python slicer and writes the
data files back to disk. No build step, no framework, no runtime deps.

---

## Run it

```
npm run editor          # node serve.mjs --open  вҶ’ opens editor.html
# or double-click launch-editor.command
# or, if a server is already up: open http://localhost:8060/editor.html
```

`serve.mjs` serves the repo on **:8060** with `Cache-Control: no-store` (Chrome's
heuristic ES-module caching otherwise runs a page on a mix of old+new modules вҖ”
impossible bugs). It's **localhost-only** and the write endpoints are dev-only.
Editing `editor.html/.js/.css` only needs a **page reload**; editing `serve.mjs`
needs a **server restart**.

## The data model вҖ” read this first

Three hand-authored sources, two generated dumps, one generated manifest. Nothing
about a map lives in one place; the editor's whole job is to keep them in step.

```
data/zones.js         GEOMETRY + identity tables  (hand-authored; SINGLE source of truth for layout)
data/placements.js    WHERE things sit on each map (GENERATED by the editor; also safe to hand-edit)
data/tile-tags.json   tile provenance + tags       (GENERATED by the slicer)
assets/tiles/*.png     the sliced tile images        (GENERATED by the slicer)
        в”Ӯ
        в”ңв”Җв–ә data/maps.txt / data/maps.json   readable dumps (regenerated on every save)
        в””в”Җв–ә src/assets-manifest.js           TILES{} keyвҶ’path map (build.mjs inlines these)
```

**`data/zones.js`** вҖ” one `ZONES` array. Each zone has `subregions[]`; each
subregion (`sub`) holds the geometry and the identity tables:

- `map: [...]` вҖ” ASCII rows, the geometry. **This is what "Paint base tile" edits.**
- `theme: 'grass'|'sand'|'swamp'|'dark'` вҖ” picks the tileset (and the editor's floor art).
- `doors: { '1': 'other-sub-id', вҖҰ }` вҖ” maps a door digit to the sub it leads to.
- `edges: { n|s|e|w: 'sub-id' }` вҖ” walk off a border floor tile into a neighbour.
- `table`, `boss`, `miniboss` вҖ” the monster pool + fixed boss/mini packs a placed
  marker *spawns*. The marker only says *where*; the identity lives here.
- `difficulty` вҖ” stands in for depth (loot scaling, exit bonus).

**`data/placements.js`** вҖ” one `PLACEMENTS` object keyed by `sub.id`:

```js
{ subId: { decor:  [{ key, x, y, w, h, rot }],   // cosmetic tiles (TILES[key])
           monsters:[{ x, y, id? }],              // id pinned, else rolls the region table
           loot:    [{ x, y, item? }],            // item pinned, else rolls gold/tome/den
           boss:    [{ x, y }], miniboss:[{ x, y }],  // spawns sub.boss / sub.miniboss
           portals: [{ x, y, to, title?, label? }] } }  // walk-onto travel trigger вҶ’ sub `to`
```

A monster/loot with **no** `id`/`item` rolls on the seeded world RNG at load (so
revisits differ); a pinned one is a deliberate authored fixture. `boss`/`miniboss`
markers spawn nothing unless the sub actually defines `boss`/`miniboss`.

### Map legend & coordinate model

`#` wall В· `.` floor В· `S` start В· `E` surface exit (banks & ends the delve) В·
`1`вҖ“`9` region door. **Doors and `E` sit ON border walls** (they're not walkable
floor); you walk *into* one from the adjacent floor tile. The loader
(`src/world/zones.js`) turns each door into `{x,y,ch,entry,dir,to}`.

- Origin top-left, `x` = column, `y` = row.
- **Decor** snaps to **half-tiles** (`w/h/rot`, free size) so props can straddle cells.
- **Markers** (monster/loot/boss/mini) snap to **whole cells**, 1Г—1.
- Every map must be **rectangular** and every floor tile **reachable from `S`** вҖ”
  the test suite enforces both (see Gotchas).

## Map mode вҖ” place & paint

The left palette has three labelled groups:

**Paint base tile** (`#tilebar`) вҖ” edits the ASCII geometry in `zones.js`.
Buttons: `Floor . / Wall # / Start S / Exit E`, plus one `Door nвҶ’dest` per entry
in `sub.doors`. Click a cell to paint it; **drag to paint a run**. Painting `Start`
moves the single `S` (clears the old one). Painted regions are flagged dirty and
written back to `zones.js` on **Save** (via `/save-map`).

**Add to map** (`#markerbar`) вҖ” the marker brushes: рҹ‘№ Monster В· рҹ’° Treasure В·
рҹ’Җ Boss В· рҹ‘ә Mini В· рҹ•іпёҸ Portal. Click a brush, then click the map to drop one. This
is how you **add** monsters/treasure (not just move/delete them). A **Portal** is a
walk-onto travel trigger (the well's Descend): drop it *under* the well/stairs art,
then set its destination region (and optional title/prompt) in the inspector вҖ”
because it's a placement now, it moves with the decor instead of orphaning.

**Decor** (`#tagfilter` + `#swatches`) вҖ” tag-filtered tile swatches. Click one to
arm it, click the map to place. New decor inherits the **last size & rotation** you
set (`deco`) вҖ” size one wall, lay a whole run without re-adjusting.

**Selecting & editing.** Click any placement to select it вҶ’ the right **inspector**
shows its controls: drag on the map, arrow-nudge, or type X/Y to move; for decor,
rotate/scale and **swap which tile it is** (the `Tile в–ҫ` dropdown вҖ” change art
without delete+re-add); for a monster/loot, **pin** an identity from the dropdown.
`Delete` removes it **immediately, no confirmation**.

**Multi-select (decorate identically).** **Shift-click** toggles a cell into a
selection; **shift-drag** sweeps a rectangle. With cells selected, clicking *any*
brush вҖ” a Paint tile, an Add-to-map marker, or a Decor swatch вҖ” stamps it into
**every** selected cell at once, then clears the selection. `Esc` clears a selection
(and disarms the brush).

**Keyboard (map mode):** `R` rotate decor В· `Del`/`Backspace` delete (no confirm) В·
arrows nudge (ВҪ tile decor / 1 cell markers) В· `[` `]` scale decor В· `Esc` deselect В·
header `вҲ’`/`+` zoom the stage.

**Save** writes `placements.js` (and regenerates `maps.txt`/`maps.json`), then, if
any geometry was painted, writes those maps back into `zones.js` in the same click.
Status line reports e.g. `Saved 14 region(s) + 1 map(s) вң“`.

**Theme-faithful floors.** Undecorated cells render the *same* art the game uses
(`#stage[data-theme]` in `editor.html` mirrors `#map[data-theme]` in `styles.css`),
so an empty grass tile is pixel-identical to placed grass decor and to the in-game
floor. If the game's per-theme tilesets change, update both blocks.

## Slice mode вҖ” cut new tiles

Pick a sheet вҶ’ **Detect objects** (outlines every non-magenta blob) вҶ’ click a box
or drag your own вҶ’ name + tag it вҶ’ **Slice**. The tile drops into the Decor palette
immediately, no reload. This drives `tools/slice_tiles.py`; the full art flow,
magenta-chroma convention, and regular-grid caveats are in
[`art-pipeline.md`](art-pipeline.md).

## Server endpoints (`serve.mjs`, dev-only)

| Route | Does |
|-------|------|
| `GET /sheets` | list `art/*.png` for the sheet picker |
| `GET /tiles` | list `assets/tiles/*` for the Decor palette (joined with `tile-tags.json`) |
| `POST /detect` | `slice_tiles.py --detect --json` on a sheet |
| `POST /slice` | `slice_tiles.py --name/--box/--tags` вҖ” one tagged tile |
| `POST /save-placements` | write `data/placements.js` + regenerate `maps.txt`/`maps.json` |
| `POST /save-map` | rewrite one or more subs' `map:[]` in `data/zones.js` (validated) + regenerate dumps |

`/save-map` body is `{ subId: [rowString, вҖҰ] }`. It **surgically replaces only the
target sub's `map[]`** (regex anchored on `id: '<sub>'` вҶ’ its `map: [ вҖҰ ]`), leaving
every comment/table/boss byte-for-byte intact, and **validates hard** before
writing: rows must be equal-length strings of `[#.SE1-9]` with **exactly one `S`**,
or it 400s and writes nothing. An identity save is byte-identical.

## Gotchas

- **`zones.js` is the one source of truth for geometry.** Paint edits go straight
  back into it. There is no separate override file вҖ” don't add one.
- **Painting can break the maps.** `test/dice.test.js` flood-fills every subregion
  (all floor must reach `S`), and asserts each sub keeps **вүҘ3 encounters** and
  **вүҘ1 loot** and that every monster/boss sits on a floor tile. Run `npm test`
  after geometry or heavy placement edits. (Heads-up: the working tree can already
  be red if a region's `loot` was emptied вҖ” e.g. `courtyard-sw` needs a Treasure.)
- **Node module cache.** `serve.mjs` statically imports `ZONES`/`PLACEMENTS` via
  `dump-map.mjs`; after a write it re-imports with `?v=<time>` (`freshZones`/
  `freshPlacements`) so the regenerated dumps reflect the just-saved geometry
  without a server restart. `dump-map`'s `renderText`/`manifests` take an optional
  `zones` arg for exactly this.
- **Triggers vs their art.** **Portals** are now an editable marker (`placements.js`
  вҶ’ `portals`), so they move with their decor вҖ” place the Portal marker under the
  well/stairs art. But **doors** (`sub.doors` + the map digit) and **edge exits**
  (`sub.edges`) are still fixed in `zones.js` and are *not* editor-editable; if you
  move a door's art, realign its `zones.js` coordinate by hand. (The SW well's
  Descend broke this way before portals became editable.)
- **`no-store` dev server** вҖ” never "fix" caching by removing it; it prevents the
  mixed-module bug.
- **Repo lives in Dropbox CloudStorage** вҖ” `git` can hang on the File Provider and
  leave a stale `index.lock`; scope git ops narrowly. See the memory note.
- **Browser-automation scaling.** A saved screenshot is scaled vs the CSS viewport,
  so screenshot pixels вү  `elementFromPoint` coordinates. Read state from the DOM
  (`getBoundingClientRect`, `.selcell`/`.obj`/`.marker` counts) rather than trusting
  screenshot-space math.

## Extending the editor

- **New base-cell char** вҶ’ add to `TILE_BRUSHES` (`editor.js`); teach the loader
  (`src/world/zones.js`) and dump (`dump-map.mjs`) what it means.
- **New theme** вҶ’ add `#stage[data-theme='x']` floor/wall rules in `editor.html`
  *and* the matching `#map[data-theme='x']` in `styles.css`.
- **New marker kind** вҶ’ extend `KINDS` / `MARKER_KINDS` / `MARKERS` in `editor.js`,
  the loader in `src/world/zones.js`, and the dump in `dump-map.mjs`.

## Key files

- `editor.html` вҖ” layout, palette, theme-faithful `#stage[data-theme]` floors, marker/paint CSS.
- `editor.js` вҖ” all editor logic: brushes (`decor`/marker/`tile`), `paintCell`,
  `stampIntoCells`, multi-select (`cellSel`/marquee), inspector, `dirtyMaps`, Save.
- `serve.mjs` вҖ” static server + editor endpoints; `rewriteZonesMap`, `freshZones`.
- `dump-map.mjs` вҖ” `renderText`/`manifests` (readable + JSON dumps; accept a `zones` override).
- `data/zones.js` В· `data/placements.js` В· `data/tile-tags.json` вҖ” the sources above.
- `src/world/zones.js` вҖ” turns a sub's ASCII map + placements into the runtime dungeon.
- `src/render/mapView.js` вҖ” how the *game* draws the same tiles/decor/loot (loot art reverts to bare floor once collected).
